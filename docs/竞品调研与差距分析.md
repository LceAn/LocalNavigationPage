# 本地导航页 — 竞品调研与功能差距分析

> 调研日期：2026-06-22
> 目的：对标市面上主流导航页 / 自托管仪表盘产品，找出本项目的功能缺口，产出**带优先级的功能清单**。
> 本项目定位：**部署在本地电脑上的个人导航页**，需同时兼顾**内网（LAN/localhost）**与**外网（公网/隧道）**两种访问路径，并清晰区分二者。

## 二、竞品核心优点提炼

### Dashy（功能最丰富，对标标杆）
- ✅ **键盘启动 / 即时搜索**：开始输入即过滤，方向键/Tab 导航，Enter 启动
- ✅ **Workspace 视图**：多应用并行切换的专门视图
- ✅ **状态检查（Status Checks）**：每个链接显示在线/离线 + 响应时间
- ✅ **丰富 Widget**：时钟、天气、搜索、RSS、动态数据等
- ✅ **多主题 + 图标包** + UI 内可视化编辑器
- ✅ **多用户 + 认证**
- ✅ **多语言**

### Homepage（轻量 + 监控强）
- ✅ **siteMonitor（站点监控）**：HTTP HEAD/GET 探测可用性 + 响应毫秒数
- ✅ **Docker/K8s 容器状态**：直接显示容器运行状态
- ✅ **100+ 服务 Widget**：Sonarr、qBittorrent、Pi-hole 等深度集成
- ✅ **完全静态、极快、全代理**

### Homarr（现代 UI + 集成）
- ✅ **拖拽式布局**（本项目的方向）
- ✅ **与自托管服务深度集成**（不只是链接，是活数据）
- ✅ **多仪表盘**（JSON 存储，可跑多套）

### Glance（多页面 + 自定义 API）
- ✅ **多页面 + 标签导航**（一套页面切换多个视图）
- ✅ **自定义 API Widget**（如 Python 脚本取 NAS 数据）

### WebStack 系列（中文生态）
- ✅ **设计精美、中文友好**
- ✅ **WebStackGo** 等提供**完整前后台 + Docker 部署**
- ✅ **分类清晰、视觉成熟**（本项目已借鉴）

### OneNav（书签管理向）
- ✅ **书签集中管理 + 导航输出**双模式
- ✅ **可视化后台管理**
- ✅ **轻量、易部署**

## 四、功能差距优先级清单

> 优先级判定标准：**对"本地电脑、内外网兼顾、个人导航页"场景的价值 × 实现可行性**。

### 🔴 P0 — 核心差异化，必做（本项目独有优势点）

#### 1. 内网 / 外网访问路径的智能区分与切换 ⭐ 用户明确需求
**现状**：项目已有"本地地址 / 隧道地址"概念雏形，但区分不清晰、无自动判断。

**目标**：
- 每个链接支持配置**内网地址**（如 `http://192.168.1.10:8080`）和**外网地址**（如 `https://nas.example.com`）
- **自动判断当前网络环境**：检测到当前页面是通过内网 IP 访问 → 默认用内网地址；通过公网域名访问 → 默认用外网地址
- **卡片上显式标注**当前用的是哪条路径（小标签 "LAN" / "公网"），并可一键切换
- 内网地址优先（更快、不走公网带宽），外网作为兜底

**实现要点**：
- 链接数据结构扩展：`{ name, url, lanUrl, wanUrl, ... }`（向后兼容，缺省回退 url）
- 环境探测：`window.location.hostname` 判断（私网段 `10./172.16-31./192.168./localhost` → 内网）
- 卡片角标：绿点 = 内网可用、蓝点 = 走公网、灰点 = 仅外网配置
- 设置里加"网络环境"说明，让用户理解自动逻辑

**为什么 P0**：这是用户**亲自提出的核心需求**，且是本项目相对其他竞品的**最强差异化点**——大多数竞品不区分内外网，而本地部署场景这恰恰是刚需。

---

#### 2. 链接在线状态监控（内网服务健康检查）
**现状**：无。

**目标**：每个内网服务链接显示**在线状态**（绿/红/灰）+ 响应时间。
- 定期（如每 60s）对内网地址做可达性探测
- 卡片上小圆点：🟢 在线（含响应 ms）/ 🔴 离线 / ⚪ 未知（外网不探）
- 失败的链接在分组顶部高亮提醒

**实现要点**（纯前端难点）：
- **纯静态限制**：浏览器无法发 ICMP ping。用 **HTTP HEAD/GET + 超时** 间接判断（Homepage 的做法）
- **跨域限制**：目标服务不开放 CORS 时，用 `<img>`/`<link>`/`fetch(no-cors)` 探测（能加载即认为可达）
- 降级：探测失败不一定离线（可能是 CORS），UI 用"未知"而非"离线"更稳妥
- **可选增强**：提供一个超轻量的 nginx 探测端点或可选的 sidecar 脚本，做真正的 TCP 探测（进阶，非 P0 必需）

**为什么 P0**：内网服务（NAS、路由器、媒体服务器）挂没挂，是本地导航页最高频的诉求，Homepage/Dashy 都以此为核心卖点。

---

### 🟠 P1 — 高价值增强

#### 3. 键盘启动器（Launcher / 命令面板升级）
**现状**：Cmd/Ctrl+K 聚焦搜索框，但需手动选引擎 + Enter。

**目标**：升级为**类 Spotlight / Raycast 的启动器**：
- 全局快捷键唤起（Cmd+K）
- 输入即过滤**所有链接**（跨分组），方向键上下选择，Enter 直接打开（用当前默认引擎或链接首选 URL）
- 支持搜索分组名、标签
- 显示最近使用优先排序

**为什么 P1**：Dashy 的核心交互，键盘党刚需，本项目已有基础（最近使用 + 搜索）。

---

#### 4. 书签 / 浏览器导入（Netscape HTML + CSV）
**现状**：仅支持 JSON 导入（项目自身格式）。

**目标**：
- 支持导入浏览器导出的 **Netscape 书签 HTML**（`bookmarks.html`，所有主流浏览器都能导出）
- 支持 **CSV**（name, url, category, tag）
- 导入时自动去重、归类、提示冲突

**为什么 P1**：降低迁移门槛——用户从浏览器书签迁过来是最大入口，OneNav/WebStack 后台版都有。

---

#### 5. 链接 / 分组拖拽排序
**现状**：搜索引擎支持拖拽，链接和分组不支持。

**目标**：
- 分组内链接拖拽排序（HTML5 drag，复用搜索引擎的 `bindSearchEngineDrag` 模式）
- 分组之间拖拽排序
- 移动端长按拖拽（touch 兼容）

**为什么 P1**：Homarr/Dashy 都支持，手动编辑顺序体验差。

---

### 🟡 P2 — 中等价值，看资源

#### 6. 多视图 / Workspace（工作 / 娱乐分页）
- 顶部切换"工作"/"娱乐"/"开发"等多套页面配置
- 参考 Glance 的多页面标签导航
- 每个视图独立的分组与链接集合

#### 7. 配置即文件（导出为可版本化的 YAML/JSON 目录）
- 当前 localStorage 难纳入 git 管理
- 提供"配置目录"导出（结构化 JSON/YAML），方便纳入版本控制、多机同步（配合 syncthing/网盘）

#### 8. 服务数据 Widget（进阶，需可选后端）
- NAS 容量、qBittorrent 下载速度、Pi-hole 拦截数等
- 由于纯静态限制，需用户提供**可选的 API 代理脚本**（参考 Glance 的 Custom API）
- 非核心，重度玩家可选装

---

### 🟢 P3 — 锦上添花，优先级最低

- **多用户 / 认证**：个人本地用需求弱（Dashy/Homarr 有，但它们面向团队 homelab）
- **RSS / 资讯订阅**：偏离导航页核心
- **多语言（i18n）**：当前中文为主，i18n 工作量大、收益看受众
- **主题市场 / 图标包导入**：Dashy 有，本项目视觉已够现代

## 六、关键洞察与建议

1. **本项目最大的差异化机会是"内外网智能区分"**：这是几乎所有竞品的盲区，而本地部署场景下是刚需。把它做透（自动判断 + 显式标注 + 一键切换 + 内网优先），能成为本项目相对 Dashy/Homepage 的独特卖点。

2. **纯静态是双刃剑**：
   - 优势：部署最简、资源最省、安全（无后端漏洞）——坚持
   - 劣势：状态监控受浏览器沙箱限制（无 ICMP ping、有 CORS）——**用探测降级（"未知"而非"离线"）+ 可选 sidecar 脚本**两条腿走

3. **不要盲目追全功能**：Dashboard 类（Widget 集成、多用户）是 Homarr/Homepage 的主场，本项目不必全面对标。**聚焦"个人、本地、内外网、美观导航"这个细分**，做精比做全更有竞争力。

4. **数据本地化是优势也要有保护**：localStorage 已有导入导出 + 备份提醒（本轮已加），继续保持；阶段二补"配置即文件"让数据可纳入 git/syncthing，解决多机同步与版本管理。
